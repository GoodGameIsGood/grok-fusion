# Screenshot Critique

Self-critique loop for visual deliverables. Soft by default; hard only on tagged visual-MVP mutate waves.

## When to run

- After first meaningful visual render
- Before claiming a UI wave done
- On redesign / polish requests

## Checklist (score mentally; revise if any fail)

1. **Brand test** — brand still readable if nav is ignored?
2. **Composition** — one first-viewport composition; hero budget respected?
3. **Anti-slop** — any banned cluster from `design-canon.md` without brief justification?
4. **Tokens** — colors/fonts reference named tokens, not random one-offs?
5. **Hierarchy** — eye path clear; one primary CTA?
6. **Clutter** — removable chrome without losing meaning?
7. **A11y** — focus, labels, contrast, reduced-motion considered?
8. **Responsive** — desktop and mobile both usable; no horizontal overflow?

## Soft vs hard

| Mode | Screenshot | Missing evidence |
|---|---|---|
| Soft (default) | Prefer browser screenshot when MCP/tools available | WARN; still list checklist results |
| Hard (visual-MVP mutate) | Required viewport evidence | BLOCK done claim |

## Rewrite protocol

1. List concrete failures (section + rule)
2. Change the fewest tokens/layout moves that fix them
3. Re-check the failed items only
4. Do not declare “best-in-class” without eval evidence
